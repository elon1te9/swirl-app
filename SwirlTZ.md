# Техническое задание на разработку приложения swirl

## 1. Общая информация

### 1.1. Название проекта

**swirl**

### 1.2. Слоган

**Круговорот знаний**

### 1.3. Тип проекта

Мобильное Android-приложение для изучения английских слов русскоязычными пользователями.

### 1.4. Цель проекта

Создать удобное, визуально приятное и понятное приложение для начинающих пользователей, которые хотят изучать английские слова через тематические разделы, уровни, изучение новых слов и закрепляющие задания.

### 1.5. Назначение приложения

Приложение должно помогать пользователю:

* изучать новые английские слова по темам;
* закреплять слова через типовые упражнения;
* проходить уровни последовательно;
* повторять уже изученные слова через ежедневный тест;
* отслеживать прогресс и серию дней обучения.

### 1.6. Целевая аудитория

Русскоязычные пользователи любого возраста, начинающие изучать английский язык с нуля.

Приложение не ориентировано на продвинутую грамматику, сложные тексты или разговорную практику. Основной фокус MVP — изучение слов.

### 1.7. Платформа

Первая версия приложения разрабатывается только под **Android**.

Публикация в Google Play на этапе MVP не планируется.

### 2.4. Хранение медиа

Для MVP рекомендуется использовать файловое хранилище на стороне backend-приложения.

Рекомендуемый подход:

* изображения и аудио хранятся на сервере в отдельных папках;
* в PostgreSQL хранятся относительные пути или публичные URL;
* API отдает изображения и аудио как статические файлы.

Пример структуры на сервере:

```text
wwwroot/
  media/
    images/
      words/
        apple.png
        doctor.png
    audio/
      words/
        apple.mp3
        doctor.mp3
    avatars/
      avatar_1.png
      avatar_2.png
```

Пример значения в БД:

```text
/media/images/words/apple.png
/media/audio/words/apple.mp3
/media/avatars/avatar_1.png
```

Такой вариант похож на Supabase Storage, но реализуется внутри собственного backend. Для учебного проекта и отсутствия бюджета это оптимальный вариант.

В будущем при необходимости можно заменить локальное хранение на S3-compatible storage, не меняя основную бизнес-логику приложения.

## 3. Общая концепция приложения

Пользователь открывает приложение, регистрируется или входит в аккаунт, попадает на главную страницу, выбирает тематический раздел и проходит уровни.

Каждый раздел содержит набор уровней. Каждый уровень содержит новые слова для изучения и набор упражнений для закрепления.

Перед прохождением заданий пользователь изучает слова уровня. После изучения всех слов он переходит к упражнениям. Если пользователь проходит все задания без ошибок, уровень считается пройденным, и открывается следующий уровень. Если пользователь допускает хотя бы одну ошибку, уровень не засчитывается, но пользователь все равно завершает текущую попытку до конца.

Отдельно в приложении есть ежедневный тест. Он формируется на основе уже изученных пользователем слов и обновляется каждый день в 00:00.

Может видеть только стартовый экран, экран входа и экран регистрации.

Может:

* войти в приложение;

* проходить ежедневный тест;

Отдельная роль администратора в приложении не реализуется.

* базу данных;
* Swagger;

## 6. Авторизация и регистрация

### 6.1. Регистрация

Пользователь должен иметь возможность зарегистрироваться через email и пароль.

Поля регистрации:

* имя пользователя;
* email;
* пароль;
* подтверждение пароля;
* аватар назначается приложением автоматически.

После успешной регистрации:

* создается пользователь;
* создается профиль пользователя;
* пользователю выдается JWT access token;
* пользователь попадает на главную страницу.

Подтверждение email в MVP не требуется.

### 6.2. Вход

Пользователь входит по email и паролю.

После успешного входа backend возвращает JWT token и данные профиля.

### 6.3. Хранение токена

На Flutter-стороне JWT token должен храниться локально в безопасном или постоянном хранилище.

Рекомендуемые варианты:

* flutter_secure_storage;
* shared_preferences для учебного MVP, если безопасность не является ключевой целью.

Для более правильной реализации рекомендуется использовать **flutter_secure_storage**.

### 6.4. Выход из аккаунта

В профиле должна быть предусмотрена возможность выйти из аккаунта.

При выходе:

* локальный JWT token удаляется;
* пользователь возвращается на экран входа.

Состояния уровня:

* заблокирован;
* доступен;
* пройден;

Экран загрузки слов или заданий.

* загрузить слова, картинки и аудио для уровня;
* подготовить упражнения.

На уровне примерно 10 новых слов. Количество слов может варьироваться.

Для каждого слова отображается:

* английское слово;
* русский перевод;
* транскрипция;
* картинка;
* кнопка прослушивания аудио;
* кнопка перехода к следующему слову.

После просмотра всех слов показывается Learned words pop-up.

Попап после изучения всех слов уровня.

Содержит:

* сообщение о завершении изучения новых слов;
* кнопку перехода к заданиям;
* возможно, кнопку вернуться к словам.

После подтверждения пользователь переходит к тренировке уровня.

### 7.13. Task with a word input

Экран задания с вводом ответа.

Варианты заданий:

* картинка → ввести английское слово;
* русское слово → ввести английский перевод;
* английское слово → ввести русский перевод.

Содержит:

* прогресс выполнения заданий;
* текст вопроса или изображение;
* поле ввода;
* кнопку проверки / продолжения;
* состояние правильного или неправильного ответа.

Проверка ответа:

* регистр не учитывается;
* пробелы в начале и конце удаляются;
* повторяющиеся пробелы внутри строки приводятся к одному пробелу;
* ответ должен совпадать с правильным ответом из БД;
* опечатки не допускаются;
* несколько вариантов правильного ответа в MVP не поддерживаются.

После ошибки правильный ответ сразу не показывается.

### 7.14. Word selection task

Экран задания с выбором ответа.

Варианты заданий:

* английское слово → выбрать русский перевод;
* русское слово → выбрать английский перевод;
* аудио английского слова → выбрать русский перевод.

Содержит:

* прогресс выполнения заданий;
* вопрос;
* варианты ответа;
* кнопку прослушивания аудио для аудио-задания;
* состояние выбранного ответа.

Количество вариантов ответа в MVP рекомендуется сделать равным 4.

Варианты ответа должны перемешиваться при каждом запуске задания.

После ошибки правильный ответ сразу не показывается.

### 7.15. Level win pop-up

Попап успешного прохождения уровня.

Показывается, если пользователь завершил все задания уровня без ошибок.

Содержит:

* сообщение об успешном прохождении;
* информацию об открытии следующего уровня;
* кнопку перехода дальше;
* кнопку возврата к карте уровней.

После успешного прохождения:

* уровень получает статус пройденного;
* следующий уровень открывается;
* слова уровня считаются изученными;
* обновляется прогресс раздела;
* обновляется streak, если это первое выполненное учебное действие за текущий день.

### 7.16. Lose level pop-up

Попап неуспешного прохождения уровня.

Показывается, если пользователь допустил хотя бы одну ошибку.

Содержит:

* сообщение о неудачной попытке;
* кнопку пройти уровень заново;
* кнопку возврата к карте уровней.

После неуспешной попытки:

* следующий уровень не открывается;
* текущий уровень не получает статус пройденного;
* ответы пользователя могут быть сохранены для истории попытки;
* изученные слова остаются изученными, если пользователь уже проходил экран изучения слов.

## 8. Структура обучения

### 8.1. Раздел

Раздел — тематическая группа уровней.

В MVP разделы:

1. Food
2. Science
3. Health
4. Wardrobe

### 8.2. Уровень

Уровень — обучающий блок внутри раздела.

Каждый уровень содержит:

* CEFR-уровень;
* набор новых слов;
* набор упражнений;
* статус доступности для пользователя.

В каждом разделе:

* 5 обычных уровней;
* 1 финальный тест.

### 8.3. Слова

В одном обычном уровне примерно 10 новых слов.

Для каждого слова нужны:

* английское слово;
* русский перевод;
* транскрипция;
* часть речи;
* картинка;
* аудио;
* привязка к уровню;
* CEFR-уровень.

### 8.4. Задания

В одном обычном уровне примерно 20 заданий.

Задания строятся на словах текущего уровня.

Финальный тест строится на словах всех 5 уровней текущего раздела.

## 9. CEFR-уровни

В приложении используются следующие уровни английского:

* A1 — Базовый;
* A2 — Легкий;
* B1 — Средний;
* B2 — Продвинутый;
* C1-C2 — Эксперт.

Приложение не определяет уровень пользователя автоматически.

Уровни контента привязываются к CEFR.

Пользователь не выбирает и не меняет свой общий уровень английского.

## 10. Типы заданий

В MVP реализуются следующие типы заданий.

### 10.1. Picture to English input

Пользователь видит картинку и должен написать английское слово.

Пример:

* картинка яблока;
* правильный ответ: `apple`.

### 10.2. English to Russian choice

Пользователь видит английское слово и выбирает русский перевод из вариантов.

Пример:

* вопрос: `apple`;
* варианты: `яблоко`, `молоко`, `хлеб`, `вода`.

### 10.3. Russian to English choice

Пользователь видит русское слово и выбирает английский перевод из вариантов.

Пример:

* вопрос: `яблоко`;
* варианты: `apple`, `milk`, `bread`, `water`.

### 10.4. Russian to English input

Пользователь видит русское слово и вводит английский перевод.

Пример:

* вопрос: `яблоко`;
* правильный ответ: `apple`.

### 10.5. English to Russian input

Пользователь видит английское слово и вводит русский перевод.

Пример:

* вопрос: `apple`;
* правильный ответ: `яблоко`.

### 10.6. Audio to Russian choice

Пользователь слушает аудио с английским словом и выбирает русский перевод.

Пример:

* аудио: `apple.mp3`;
* варианты: `яблоко`, `молоко`, `хлеб`, `вода`.

## 11. Логика прохождения уровня

### 11.1. Старт уровня

Пользователь выбирает доступный уровень на карте.

Если слова уровня еще не изучены, пользователь сначала проходит этап изучения слов.

Если слова уже изучены, пользователь может сразу начать тренировку.

### 11.2. Изучение слов

Пользователь последовательно просматривает все слова уровня.

После просмотра всех слов:

* слова считаются просмотренными;
* пользователь может перейти к заданиям.

### 11.3. Выполнение заданий

Пользователь выполняет все задания уровня последовательно.

Если пользователь допускает ошибку:

* ошибка фиксируется;
* правильный ответ сразу не показывается;
* пользователь продолжает проходить уровень до конца.

### 11.4. Завершение уровня

После последнего задания backend или приложение подсчитывает количество ошибок.

Если ошибок 0:

* уровень считается пройденным;
* открывается следующий уровень;
* показывается Level win pop-up.

Если ошибок больше 0:

* уровень не засчитывается;
* следующий уровень не открывается;
* показывается Lose level pop-up.

### 11.5. Сохранение незавершенного уровня

Если пользователь вышел из уровня или закрыл приложение, прогресс текущей попытки должен сохраняться.

Для MVP допустимы два варианта:

1. сохранять незавершенную попытку локально во Flutter;
2. сохранять незавершенную попытку на backend.

Рекомендуемый вариант для MVP:

* состояние текущей попытки хранить локально;
* на backend отправлять результат только после завершения уровня.

Так проще и быстрее в реализации.

Однако, если важно продолжать уровень после входа с другого устройства, нужно хранить attempt на backend. Для MVP это не требуется.

---

## 12. Финальный тест раздела

Финальный тест открывается после успешного прохождения всех 5 уровней раздела.

Особенности:

* новых слов в финальном тесте нет;
* задания строятся на словах всех уровней текущего раздела;
* тест считается пройденным только при 0 ошибках;
* при ошибках финальный тест нужно проходить заново;
* прохождение финального теста увеличивает общий прогресс раздела.

---

## 13. Ежедневный тест

### 13.1. Назначение

Ежедневный тест нужен для повторения уже изученных пользователем слов.

### 13.2. Расположение

На главной странице должна быть отдельная карточка ежедневного теста.

### 13.3. Условия доступности

Ежедневный тест доступен, если у пользователя есть хотя бы минимальное количество изученных слов.

Рекомендуемое условие:

* если изучено меньше 5 слов — карточка отображается как недоступная или показывает текст “Изучите больше слов, чтобы открыть ежедневный тест”.

### 13.4. Формирование теста

Тест формируется на основе уже изученных слов пользователя.

Количество заданий:

* минимум 15;
* максимум 30.

Если изученных слов мало, количество заданий может быть меньше 15.

Слова выбираются из всех изученных разделов в случайном порядке.

### 13.5. Обновление

Ежедневный тест обновляется каждый день в 00:00.

Для MVP рекомендуется считать день по серверному времени.

### 13.6. Прохождение

Ежедневный тест может использовать те же типы заданий, что и обычный уровень.

Результат ежедневного теста не должен блокировать прогресс по разделам.

### 13.7. Streak

Если пользователь в течение дня успешно выполнил хотя бы одно учебное действие, streak обновляется.

Учебными действиями считаются:

* успешное прохождение уровня;
* попытка прохождения уровня;
* прохождение ежедневного теста.

Рекомендуемая логика для MVP:

* streak обновляется один раз в день при завершении любого уровня или ежедневного теста;
* если пользователь занимался вчера и занимается сегодня — streak увеличивается на 1;
* если пользователь пропустил день — streak сбрасывается до 1;
* если пользователь уже занимался сегодня — streak не увеличивается повторно.

## 14. Прогресс пользователя

### 14.1. Что нужно хранить

Для каждого пользователя нужно хранить:

* пройденные уровни;
* открытые уровни;
* изученные слова;
* ответы пользователя;
* статус ответов;
* попытки прохождения уровней;
* streak;
* лучший streak;
* дату последней активности;
* прогресс по разделам.

### 14.2. Прогресс раздела

Прогресс раздела считается на основе количества пройденных уровней и финального теста.

Если в разделе 5 уровней и 1 финальный тест, всего 6 элементов прогресса.

Формула:

```text
progress = completed_items / total_items * 100
```

Пример:

* пройдено 3 уровня из 6 элементов;
* прогресс = 50%.

### 14.3. Изученные слова

Слово считается изученным, если пользователь просмотрел его на этапе Learn word.

Слово может использоваться в ежедневном тесте только после того, как оно стало изученным.

### 14.4. Ответы пользователя

Для каждого ответа рекомендуется хранить:

* пользователя;
* attempt;
* exercise;
* ответ пользователя;
* правильность;
* дату ответа;
* время ответа, если нужно.

## 15. Медиа: изображения, аудио, аватары

### 15.1. Изображения слов

Картинки слов должны быть реальными изображениями.

Для каждого слова желательно иметь отдельную картинку.

Backend хранит путь к картинке в таблице `words`.

Пример:

```text
/media/images/words/apple.png
```

Flutter получает путь от API и загружает изображение по URL backend-сервера.

### 15.2. Аудио слов

Для каждого слова желательно иметь аудио с произношением английского слова.

Backend хранит путь к аудиофайлу.

Пример:

```text
/media/audio/words/apple.mp3
```

Flutter получает путь и проигрывает аудио через аудио-плеер.

### 15.3. Аватары

В MVP пользователь получает аватар из готовых вариантов автоматически при регистрации.
Позже пользователь сможет сменить аватар в профиле или настройках.

Аватары хранятся на backend в папке:

```text
/media/avatars/
```

В БД у пользователя хранится `avatarId` или `avatarUrl`.

Рекомендуемый вариант:

* создать таблицу `avatars`;
* хранить в профиле пользователя `avatar_id`.

Это лучше, чем хранить строку URL напрямую в профиле, потому что позже можно будет менять путь к файлу без изменения всех пользователей.

### 15.4. Почему медиа лучше хранить на backend, а не локально в приложении

Для swirl лучше хранить картинки и аудио на backend, потому что:

* контент можно менять без пересборки Flutter-приложения;
* можно добавлять новые слова и уровни через БД/API;
* структура похожа на Supabase Storage;
* легче расширять приложение;
* приложение не будет раздуваться из-за большого количества картинок и аудио.

Для учебного проекта это идеальный баланс между простотой и правильной архитектурой.

## 18. Генерация заданий

### 18.1. Обычный уровень

Для каждого уровня можно заранее создать упражнения в базе.

Рекомендуемый вариант для MVP:

* слова хранятся в таблице `words`;
* задания создаются заранее в таблице `exercises`;
* варианты ответов создаются заранее в `exercise_options`.

Это упростит реализацию и сделает результат предсказуемым.

### 18.2. Варианты ответов

Для заданий с выбором должно быть 4 варианта ответа:

* 1 правильный;
* 3 неправильных.

Неправильные варианты можно брать из слов того же раздела или уровня.

Для MVP лучше хранить варианты в БД, чтобы не писать сложную генерацию.

### 18.3. Ежедневный тест

Ежедневный тест лучше генерировать динамически.

Алгоритм:

1. Получить все изученные пользователем слова.
2. Если слов меньше 5 — тест недоступен.
3. Перемешать слова.
4. Выбрать от 15 до 30 слов.
5. Для каждого слова случайно выбрать тип задания из доступных.
6. Для choice-заданий сформировать варианты ответа.
7. Вернуть тест во Flutter.

## 19. Проверка ответов

### 19.1. Нормализация ответа

Перед проверкой ответ нужно нормализовать:

* удалить пробелы в начале и конце;
* привести строку к нижнему регистру;
* заменить повторяющиеся пробелы одним пробелом.

Пример:

```text
"  APPLE  " → "apple"
"  green   apple " → "green apple"
```

### 19.2. Правила проверки

Ответ считается правильным, если нормализованный ответ пользователя полностью совпадает с нормализованным правильным ответом из БД.

Опечатки не принимаются.

Несколько правильных вариантов в MVP не поддерживаются.

### 19.3. Поведение после ответа

После ответа пользователь видит состояние:

* верно;
* неверно.

Правильный ответ после ошибки не показывается.

## 24. Нефункциональные требования

### 24.1. Производительность

Приложение не должно отправлять отдельный запрос на каждое задание.

Правильная логика:

1. Пользователь запускает уровень.
2. Flutter одним запросом получает всю сессию уровня.
3. Пользователь проходит задания локально.
4. Flutter отправляет результат после завершения уровня.

Это сделает приложение более быстрым и устойчивым к задержкам сети.

### 24.2. Надежность

Если во время загрузки данных произошла ошибка, пользователь должен увидеть понятное сообщение и кнопку повторить.

### 24.3. Безопасность

* пользователь может видеть только свой прогресс;
* все пользовательские endpoints должны быть защищены JWT;
* пароль хранится только в виде хэша;
* backend не должен возвращать password_hash;
* пользователь не может изменять прогресс другого пользователя.

### 24.4. Масштабируемость

Архитектура должна позволять позже добавить:

* новые разделы;
* новые уровни;
* новые типы заданий;
* загрузку аватара;
* админ-панель;
* S3-хранилище;
* iOS-версию;
* монетизацию.

## 29. Важные технические решения

### 29.1. Почему выбран собственный backend

Для проекта выбран ASP.NET Core Web API + PostgreSQL, потому что:

* приложение должно стабильно работать в текущих сетевых условиях;
* есть необходимость контролировать backend-логику;
* можно гибко управлять прогрессом, уровнями и заданиями;
* легче реализовать собственную логику ежедневного теста;
* проект является учебным и полезен для портфолио.

### 29.2. Почему не Supabase

Supabase удобен для быстрого старта, но в данном проекте уже был негативный опыт с задержками и нестабильной загрузкой. Для приложения с быстрыми учебными заданиями отзывчивость важна, поэтому собственный API является более контролируемым решением.

### 29.3. Почему медиа на backend

Медиа на backend позволяет:

* менять картинки и аудио без пересборки приложения;
* хранить в БД только ссылки;
* работать по модели, похожей на Supabase Storage;
* в будущем перейти на S3 без изменения логики Flutter.

## 31. Итог

swirl — это Android-приложение для русскоязычных начинающих пользователей, изучающих английские слова через тематические разделы, уровни, изучение слов и закрепляющие задания.

Основной технический принцип проекта: приложение должно получать данные уровня пакетно, проходить задания без постоянных запросов к серверу и отправлять результат после завершения уровня. Это обеспечит хорошую скорость работы и удобный пользовательский опыт.
